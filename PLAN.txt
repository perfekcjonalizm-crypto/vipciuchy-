WYBRANY STOS TECHNOLOGICZNY
-----------------------------
- Runtime:   Python 3.9 (dostępny natywnie na macOS)
- Backend:   Flask 3.1 (zainstalowany przez pip)
- Baza:      SQLite (wbudowany w Python, plik: backend/rzeczy.db)
- CORS:      flask-cors 6.0
- Auth:      Flask sessions + werkzeug.security (pbkdf2:sha256)
- Frontend:  istniejący HTML z fetch() do http://localhost:5000/api/

UWAGA — pbkdf2 zamiast scrypt:
  Python 3.9 z CommandLineTools nie posiada hashlib.scrypt.
  Używamy pbkdf2:sha256 — bezpieczne, w pełni obsługiwane.

SCHEMAT BAZY DANYCH
--------------------
  users      — id, username, email, password_hash, avatar, created_at
  products   — id, name, brand, price, size, condition, emoji,
               description, seller_id, is_sold, created_at
  favorites  — user_id, product_id (PK złożony)
  orders     — id, product_id, buyer_id, seller_id, amount,
               platform_fee, seller_amount, payment_method,
               status, created_at
  messages   — id, from_user_id, to_user_id, product_id,
               content, created_at

ENDPOINTY API (wszystkie: http://localhost:5000)
-------------------------------------------------
  GET  /api/health                 — healthcheck
  GET  /api/products               — lista (?q=, ?sort=, ?page=)
  GET  /api/products/<id>          — szczegóły produktu
  POST /api/products               — dodaj produkt (auth)
  POST /api/products/<id>/favorite — toggle ulubione (auth)
  GET  /api/products/favorites     — moje ulubione (auth)
  POST /api/auth/register          — rejestracja
  POST /api/auth/login             — logowanie
  POST /api/auth/logout            — wylogowanie
  GET  /api/auth/me                — aktualny użytkownik
  POST /api/orders                 — złóż zamówienie (auth)
  GET  /api/orders/<id>            — szczegóły zamówienia (auth)
  GET  /api/orders/my              — moje zamówienia (auth)
  GET  /api/messages/<product_id>  — historia czatu (auth)
  POST /api/messages               — wyślij wiadomość (auth)

INTEGRACJA FRONTENDU
---------------------
  - Warstwa API działa w trybie "progresywnym":
    * Backend OFFLINE → aplikacja działa jak dotychczas (PRODS[])
    * Backend ONLINE  → PRODS[] zastąpiony danymi z bazy,
      logowanie/rejestracja, czat i zamówienia trafiają do API
  - Zmienione buttony:
    * "Zaloguj się"         → doLogin()
    * "Utwórz konto"        → doRegister()
    * "Opublikuj ogłoszenie"→ doSell()
    * "Potwierdź BLIK"      → confirmBlik() → POST /api/orders
    * sendMsg()             → POST /api/messages (z auto-reply z bazy)

WYNIKI AUDYTU
--------------
  ✓ SQL injection — wszystkie zapytania używają parametryzowanych
    placeholders (?, nie f-string/format)
  ✓ XSS — esc() w HTML; API zwraca JSON (nie HTML)
  ✓ CORS — dozwolone tylko localhost:5000 i file://
  ✓ Hasła — pbkdf2:sha256 (werkzeug), nigdy plaintext
  ✓ Autoryzacja — endpointy modyfikujące wymagają session.user_id
  ✓ Własny produkt — nie można kupić swojego ogłoszenia
  ✓ Produkt sprzedany — nie można zamówić is_sold=1
  ✓ Foreign keys — PRAGMA foreign_keys = ON

WYMAGANA UWAGA UŻYTKOWNIKA
---------------------------
  1. secret_key w app.py: os.urandom(32) — po każdym restarcie
     serwera sesje są invalidowane (użytkownicy muszą się ponownie
     zalogować). W produkcji ustaw stały klucz z os.environ.
     → Zmień w app.py: app.secret_key = 'twoj-staly-sekretny-klucz'

  2. Upload zdjęć nie jest zaimplementowany (backend nie obsługuje
     plików multipart). Przycisk w "Sprzedaj" nadal pokazuje demo.
     → Wymaga dodania endpointu POST /api/upload i obsługi storage.

  3. Płatności Stripe (karta, Google Pay, Apple Pay) nadal wskazują
     na https://stripe.com — to wymaga integracji Stripe API (klucze).
     Tylko BLIK (mock) jest podłączony do lokalnych zamówień.
